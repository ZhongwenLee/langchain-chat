# 架构说明

## 配置体系

项目采用“基础配置 + 环境覆盖 + 进程变量最终覆盖”的三层模式：

1. `config.yaml` / `.env` 作为默认基线
2. `config.{env}.yaml` / `.env.{env}` 提供环境差异化覆盖
3. 运行时注入的环境变量具有最高优先级

当前支持的环境名为 `dev`、`test`、`prod`，通过 `APP_ENV` 切换。

### 设计目标

- 开发、测试、生产互不污染
- 切换环境无需修改代码
- 配置文件尽量只写差异项，减少重复
- 敏感信息按环境隔离，避免本地、CI 和生产共用同一套密钥

### 合并规则

- 字典会递归深度合并
- 列表采用整项覆盖
- 缺失的环境专属文件不会报错，系统会自动回退到基础文件

### 数据库隔离

每个环境都应使用独立的数据库 URL。
对于 SQLite，本地开发可使用单独文件，测试建议使用内存库或独立临时文件，生产则使用独立持久化实例。

### 密钥隔离

不同环境必须使用不同的 `API_KEY`。
测试环境应使用专用测试密钥，生产环境应使用部署平台注入的正式密钥。
